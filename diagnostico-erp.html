2 Arquitectura actual

Routing / navegación

No existe ningún router instalado (react-router, wouter, etc.). La navegación es un useState("dashboard") en App.jsx que acepta los valores:

"dashboard" | "form" | "view" | "config" | "clientes"

Cada pantalla se monta y desmonta mediante un bloque condicional {screen === "x" && <PantallaX />}. No hay URLs por pantalla, no hay historial del navegador, y el botón atrás del navegador sale de la app en lugar de volver a la pantalla anterior.

Estado global

No hay Context, Zustand, Redux ni gestor de estado externo. Todo el estado vive en App.jsx como variables useState:

VariableContenido
sessionSesión de Supabase Auth
configFila única de la tabla config (datos empresa)
solicitudesArray completo de solicitudes en memoria
clientesArray completo de clientes en memoria
screenPantalla activa (string)
editing / viewingSolicitud seleccionada actualmente

El patrón es prop drilling puro: App baja datos y callbacks a cada screen; las screens llaman los handlers y App actualiza el estado central.

Responsabilidades de src/lib/

ArchivoResponsabilidad

4 Acoplamiento

Grado de acoplamiento actual

Alto. El código de solicitudes está fuertemente acoplado al resto por la arquitectura de orquestación centralizada en App.jsx.

App.jsx es un monolito de orquestación. Gestiona simultáneamente autenticación, estado de solicitudes, estado de clientes, estado de config y navegación. Tiene 17 funciones handler y carga los tres recursos en el mismo useEffect.

La navegación por string literal es frágil. Los valores de pantalla están escritos directamente como strings dispersos en App.jsx. No hay un sitio central de declaración de rutas.

El objeto ESTADOS está duplicado entre DashboardScreen.jsx y ViewScreen.jsx. Añadir un estado nuevo requiere cambiar dos archivos.

ConfigScreen.jsx accede a Supabase directamente (define su propia función dbSaveConfig local en línea 21), duplicando la que ya existe en db.js.

¿Es fácil añadir módulos nuevos?

Añadir un módulo (Flota, Facturación, Partes de trabajo) requiere hoy tocar obligatoriamente:

ArchivoQué hay que tocar
App.jsxAñadir useState, handlers y bloque condicional de render
App.jsx línea 38Ampliar el Promise.all de carga inicial
screens/index.jsExportar la nueva pantalla
lib/db.jsAñadir nuevas funciones de acceso (esto sí es limpio)

Lo que no requeriría tocar: lib/pdf.js, lib/messaging.js, los componentes UI.

5 Preparación para crecer

¿La estructura aguanta convertirse en un ERP multi-módulo?

No directamente. La arquitectura es correcta para una app de una entidad gestionada por una persona. Para un ERP multi-módulo encontrará estos límites concretos:
LímiteDescripción
App.jsx como "dios" Con 4–5 módulos, tendría ~50 useState, ~40 handlers y ~15 bloques condicionales de render. Inmanejable.
Sin router Sin URLs por pantalla no hay posibilidad de compartir enlaces, bookmarks ni navegación con historial. Bloqueador real para un ERP.
Carga de datos monolítica Todo se carga al login. Para un ERP con flota, facturas, empleados, etc., esto sería inmanejable en rendimiento y memoria.
Sin roles de usuario Supabase Auth está integrado pero solo hay un nivel de acceso. Un ERP necesita roles (comercial, administración, operario).

Reorganización de carpetas recomendada

src/ ├── modules/ │ ├── solicitudes/ ← lo que existe hoy │ │ ├── screens/ │ │ ├── db.js │ │ └── index.js │ ├── clientes/ ← mover de config → módulo propio │ ├── flota/ ← nuevo │ ├── facturacion/ ← nuevo │ └── partes/ ← nuevo ├── shared/ │ ├── components/ui/ ← lo que existe hoy │ ├── lib/ ← utils, supabase client │ └── hooks/ ├── router/ ← nueva capa └── App.jsx ← solo layout y auth

Riesgos concretos de romper solicitudes al ampliar

RiesgoDescripción
Refactor de estado global Si se migra a Context o Zustand, los handlers de solicitudes pueden perder sincronización con viewing y editing.
Campo vehiculo como CSV Si se migra a array real en BBDD, el código de serialización en db.js hay que actualizar en sincronía con la migración de datos.
fotos en jsonb de solicitudes Las fotos se guardan como jsonb en la propia fila. Si la tabla crece, esto penaliza los listados que hacen select *.
config fila única con id=1 Antipatrón para multi-empresa o multi-sucursal. Requiere migración si se necesita multi-tenant.
notas_seguimiento como jsonb array No se puede buscar ni filtrar por notas desde SQL de forma eficiente. Candidato a tabla propia si el volumen crece.
fecha como string DD/MM/YYYY No es un tipo date en BBDD. Impide filtros por rango de fecha en SQL. Requiere migración de datos.
ESTADOS duplicado Cambio de estados rompe si no se actualiza en DashboardScreen.jsx y ViewScreen.jsx simultáneamente.

Resumen ejecutivo

La app tiene una base de código limpia, legible y funcional para su propósito actual (~2.300 líneas totales, 3 tablas, sin dependencias pesadas de UI).

El cuello de botella para crecer a ERP no está en ningún archivo concreto sino en dos puntos arquitecturales: la orquestación centralizada en App.jsx (estado global monolítico) y la ausencia de un router (navegación por string sin URLs).

Esos dos puntos son los que deben abordarse antes de añadir cualquier módulo nuevo, para evitar deuda técnica inmanejable desde el primer módulo adicional. El resto de la lógica (db.js, lib/, componentes UI) está bien aislado y reutilizable tal como está.
